# Skill: 通信故障通用原理

## 适用范围

所有具有内部通信总线的机器人(以太网、CAN、RS485、内部 PCIe 等)。

## 通信故障的三层模型

任何通信问题可拆为三层:

```
┌─────────────────────────────┐
│   应用层(数据语义)          │  字段值是否正确、序列化是否一致
├─────────────────────────────┤
│   协议层(消息传递)          │  延迟、丢包、QoS、拓扑发现
├─────────────────────────────┤
│   物理层(连接)              │  线缆、接口、电气信号
└─────────────────────────────┘
```

**永远从下往上排查**。物理层有问题,上层任何调优都无效。

## 三步诊断方法

### 第一步:物理层(基础)

物理层问题征兆:
- 完全无数据
- 间歇性丢失
- 数据严重损坏(CRC 错误)

物理层检查:
- 线缆是否插紧?
- 接口是否氧化、变形?
- 链路指示灯是否正常?
- 是否有外部干扰源(强电磁、高功率电机)?

物理层处置:
- 重插
- 换线
- 隔离干扰源
- 检查接地

### 第二步:协议层(性能)

协议层问题征兆:
- 数据可达但延迟高
- 偶尔丢包(< 5%)
- 频率不稳

协议层检查:
- 带宽占用是否过载?
- QoS 配置是否合理?
- 拓扑是否有环路或多路径?
- 缓冲区是否溢出?

协议层处置:
- 降低高频 topic 频率
- 调整 QoS(reliable vs best_effort)
- 优化拓扑
- 调大缓冲区

### 第三步:应用层(语义)

应用层问题征兆:
- 数据可达且及时,但内容异常
- 字段值看起来不对
- 偶发解析错误

应用层检查:
- 序列化版本是否匹配?
- 字段定义是否一致?
- 字节序是否一致?

应用层处置:
- 升级所有节点到一致版本
- 检查 msg 定义文件版本

## 通信问题的间接影响

通信不稳会**伪装成**其他问题,常见误诊:

| 表面现象 | 真实原因 | 误诊为 |
|---|---|---|
| 关节响应慢 | 通信延迟 | 关节机械问题 |
| 平衡控制失效 | IMU 数据丢包 | IMU 传感器坏 |
| 命令无效 | 服务超时 | 命令本身错 |
| 数据漂移 | 时间戳不同步 | 估计器漂移 |

诊断其他子系统问题时,**先确认通信健康**。

## 关键监控指标

| 指标 | 健康阈值 | 警告 | 严重 |
|---|---|---|---|
| 关键控制 topic 延迟 | < 10ms | > 30ms | > 100ms |
| 丢包率 | < 0.1% | > 1% | > 5% |
| 心跳间隔(主从) | 10-20ms | > 30ms | > 100ms |
| 主总线带宽占用 | < 50% | > 70% | > 90% |

## 不在本 skill 范围

- 具体的 ROS topic / DDS 配置
- 具体的网卡/IP/端口
- 具体的 SDK API
- 具体的诊断命令
