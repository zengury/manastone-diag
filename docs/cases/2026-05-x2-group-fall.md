## 三、三台机器诊断详情

### 3.1 X2-A — 软件崩溃型(BMIMIC 推理超限)

| 项目 | 内容 |
|------|------|
| **版本** | v1.0.0.23-bvlost-rc2(测试版) |
| **摔倒时刻** | 2026-05-25 09:29:55 |
| **摔倒类型** | Sustained Bad Posture(姿态缓慢崩溃,历时约 8 秒) |
| **摔倒时模式** | BMIMIC(身体模仿,ONNX 驱动) |
| **触发来源** | 群控经 app_proxy 下发 SetMcMotion |

**根因**:

```
rl_bmimic_t0 task run time: 58.591ms, period: 20ms, miss 192.955%
```

BMIMIC 首次 ONNX 推理在第一个控制周期就跑了 **58.6ms**(约 3 个周期),
后续关节指令持续积压,8 秒内躯干从直立偏至约 89°,触发 Sustained Bad
Posture 保护性摔倒。

**证据链**:

```
09:29:47  BMIMIC 被触发(群控下发 SetMcMotion, MIMIC 类型)
09:29:47  rl_bmimic_t0 超限 58.6ms  ← 第一拍就超
09:29:47  关节指令发布延迟 2–3.5ms
09:29:50  关节指令发布持续延迟
09:29:53  EtherCAT 出现 timing 抖动  ← 晚 6 秒,是果不是因
09:29:55  躯干角 1.56 rad → Sustained Bad Posture → DAMPING(安全响应 0.05s)
```

**与厂家初判的分歧**:日志中无 EtherCAT link-down、无 DC 校准错误;
时序上任务超限早于 EtherCAT 告警 6 秒。"通信中断"解释不成立。

**建议**:检查 v1.0 ONNX 模型在 DCU v2.0.5 + 电机 PFMotor v2.1.0
组合下的推理性能;MC 侧增加任务超限保护。

### 3.2 X2-B — 硬件通信型(EtherCAT DC 校准失效)

| 项目 | 内容 |
|------|------|
| **版本** | v0.9.7 |
| **摔倒时刻** | 2026-05-25 09:22:06 |
| **摔倒类型** | Fall from move state(运动中摔倒,高角速度,1.3 秒) |
| **摔倒时模式** | LOCOMOTION_DEFAULT(CPGWalk) |
| **现场描述** | "刚走一步就倒地、缓慢倒了、后有电机异响" |

**根因**:EtherCAT DC(Distributed Clock)校准在行走启动瞬间失效,
多电机同步崩塌。

**独有证据**:

```
09:22:05.209  DC calibration error detected, enabling network monitor
09:22:05.209  Frame tick over 2ms, diff: 3,440,458 ns (3.4ms)
09:22:05.212  ethercat master loop over dc clock: 3,985,320 ns (4ms)
              → 持续约 80ms 的 EtherCAT 同步雪崩
09:22:06.191  摔倒(仅 1.3 秒后)
```

**"电机异响"的解释**:摔倒后安全模块进入异常循环 ——

```
09:22:06.352  安全等级 L0(紧急停机)
09:22:06.717  DAMPING 结束
09:22:07.364  安全等级升至 L1
09:22:07.515  "Will not respond low level secure level: L1"
              → 此后每 100ms 循环一次,持续 2+ 秒
              → 电机反复在卸力/锁紧间切换 → 异响
```

**与厂家初判的关系**:部分一致 —— 确实存在 EtherCAT 通信异常,
但性质是"DC 时钟失步"而非"链路中断"。**建议本台返厂,重点检查
EtherCAT 主站时钟源与从站 DC 配置。**

### 3.3 X2-C — 软件崩溃型(animation_player 同步阻塞)

| 项目 | 内容 |
|------|------|
| **版本** | v0.9.7 |
| **摔倒次数** | **2 次**(起立时 + 行走时,间隔 6 分钟) |

**根因**:`animation_player` 在 RT 线程中同步执行姿态规划
(`PlanningArmPos`),单次阻塞 **484–643ms**。

**第一摔(起立时,2026-05-22 13:53:27)**:

```
13:53:23  LIE_UP_DEFAULT 转换开始
13:53:26  rl_play_t0 task run time: 2879.82ms, miss 14299.1%
13:53:27  Detect Falling Action(1.3 秒后)
```

**第二摔(行走时,13:59:30)**:

```
13:59:20  animation_player 运行 490.956ms, miss 24447.8%
13:59:28  LOCOMOTION_DEFAULT 转换
13:59:30  Fall detected(躯干仅 0.59 rad → 小角度即触发)
```

**与厂家初判的分歧**:该机 EtherCAT 告警虽多达 1670 条,但两次摔倒
时刻附近**均无 DC 校准错误**,第一次摔倒时 EtherCAT 完全正常。
根因是 RT 任务阻塞。

## 五、共性缺陷:animation_player 同步阻塞

三台机器的共同特征是**模式切换期间 `animation_player` 在 2ms 周期的
硬实时线程中同步执行姿态规划**,单次阻塞 484–643ms(超限 ~24000%)。
即使某次超限没有直接导致摔倒,也大幅压缩了安全裕度 —— 这是架构层面
的系统性缺陷,不是偶发硬件坏线。

## 六、本次诊断暴露的能力盲区

| 盲区 | 影响 |
|------|------|
| 关节状态 topic 未在 MCAP 记录 | 无法回溯关节实际位置/速度/力矩时序 |
| 胸部 IMU 未在 MCAP 记录 | 无法回溯完整姿态倾角过程 |
| HDS 诊断不覆盖摔倒检测 | 无独立诊断码佐证 |
| X2-C 使用两个 log session | ingestor 首轮漏报第一次摔倒 |

这四条当场写回了 `capability_boundary.yaml` 与 `interfaces.yaml`
(新增 3 个 topic + 1 个 service)—— 下次诊断自动受益。

## 七、结论与建议

| 厂家初判 | 诊断结论 |
|----------|---------|
| 三台都是"通信链路中断" | 仅 X2-B 涉及通信硬件问题(DC 失步);X2-A 与 X2-C 为软件缺陷 |
| 三台都需返厂 | X2-B 返厂检查 EtherCAT 时钟硬件;X2-A、X2-C 应走软件修复 |

| 优先级 | 措施 | 适用 |
|--------|------|------|
| **P0** | `animation_player` 姿态规划剥离 RT 线程,改异步或预计算 | 全部 |
| **P0** | 所有 RT 任务加 watchdog:连续超限 >5 倍自动切 DAMPING | 全部 |
| **P0** | X2-B 单独返厂,检查 EtherCAT 主站时钟源与从站 DC 配置 | X2-B |
| **P1** | X2-A 单独 benchmark BMIMIC ONNX 推理耗时 | X2-A |
| **P1** | `rl_play`(起立)加 timeout,单次不超过 200ms | 全部 |
| **P2** | 关节状态 topic 加入 MCAP 录制配置(修复盲区) | 全部 |
